1 ## MCStepLogger
2 
3 Detailed debug information about stepping can be directed to standard output using the `LD_PRELOAD` env variable, which "injects" a
4  special logging library (which intercepts some calls) in the executable that follows in the command line.
5 
6 
7 ~~~~~~~{.sh}
8 
9 LD_PRELOAD=path_to/libMCStepLogger.so o2sim -m MCH -n 10
10 ~~~~~~~

13 ~~~~~~~

15 [STEPLOGGER]: did 28 steps
16 [STEPLOGGER]: transported 1 different tracks
17 [STEPLOGGER]: transported 1 different types
18 [STEPLOGGER]: VolName cave COUNT 23 SECONDARIES 0
19 [STEPLOGGER]: VolName normalPCB1 COUNT 3 SECONDARIES 0
20 [STEPLOGGER]: ----- END OF EVENT ------
21 [FIELDLOGGER]: did 21 steps
22 [FIELDLOGGER]: VolName cave COUNT 20
23 [FIELDLOGGER]: ----- END OF EVENT ------

27 The stepping logger information can also be directed to an output tree for more detailed investigations.
28 Default name is `MCStepLoggerOutput.root` (and can be changed
29 by setting the `MCSTEPLOG_OUTFILE` env variable).
30 
31 
32 ~~~~~~~{.sh}
33 
34 MCSTEPLOG_TTREE=1 LD_PRELOAD=path_to/libMCStepLogger.so o2sim ..
35 ~~~~~~~
36 
37 Finally the logger can use a map file to give names to some logical grouping of volumes. For instance to map all sensitive volumes from a given detector `DET` to a common label `DET`. That label can then be used to query information about the detector steps "as a whole" when using the `StepLoggerTree` output tree.
38 
39 
40 ~~~~~~~{.sh}
41 
42 > cat volmapfile.dat
43 normalPCB1 MCH
44 normalPCB2 MCH
45 normalPCB3 MCH
46 normalPCB4 MCH
47 normalPCB5 MCH
48 normalPCB6 MCH
49 centralPCB MCH
50 downroundedPCB MCH
51 uproundedPCB MCH
52 cave TheCavern
53 
54 > MCSTEPLOG_VOLMAPFILE=path_to_/volmapfile.dat MCSTEPLOG_TTREE=1 LD_PRELOAD=path_to/libMCStepLogger.so o2sim ..
55 
56 > root -b MCStepLoggerOutput.root
57 root[0] StepLoggerTree->Draw("Lookups.volidtomodule.data()");
58 ~~~~~~~
